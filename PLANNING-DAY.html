Planning Day — the working checklist

Everything your team must decide before build day, beyond the feature itself. Leave at 5pm with every DECIDE answered in writing, every WHO a named person, and your committed scope locked. Build day runs on these answers.

AThe codebase plan fits the 1:30–2:30 work-breakdown block

  • MUSTMap the god-files and make them lanes. Legacy CF routes a lot through shared monsters (engine.cfm variants, Application.cfc, shared components). List the ones your feature touches; each becomes a SERIALIZED lane — one ticket at a time, one owner. Parallel agents editing the same 5,000-line file is the #1 merge disaster.
  • MUSTSerialize database changes. Agents are read-only on the dev DB (repo rule — SELECT only). Schema changes are written as scripts in db_scripts/ and a HUMAN applies them. Name who applies them and batch when — two tickets altering the same tables collide invisibly to git. WHO one DB-script owner per team.
  • DECIDEScaffolding-first sequencing: what must exist (schema, contracts, shared components, fixtures) before parallel lanes branch? Who builds it first thing at 8:15?
  • MUSTExit criterion: every person leaves planning day with ≥1 claimable Todo ticket that does NOT depend on scaffolding (contracts, fixtures, test harness, seed data all count).
  • DECIDEFrontend/backend split: React app work (v1/views — builds and tests locally per worktree) vs CFML work (verifiable only against the dev environment). Weight assignments accordingly — frontend lanes parallelize safely, CFML lanes need the merge-to-verify loop.

BThe verification plan decide BEFORE the day ends — the done-gate depends on it

The single most important decision of planning day: what is your team's testCommand? swb done refuses to finish a ticket unless this command passes. If you don't define a real one, your done-gate is theater and the rubric's 15% automated-testing line is lost. There is no TestBox harness in the repo today.
  • DECIDEOptions, honestly ranked: (1) frontend — the v1/views build/lint/jest commands are real gates and exist today; (2) CFML — a smoke script (curl key endpoints on the dev instance, grep for errors) is a legitimate gate QA can own; (3) TestBox if someone gets it standing during planning — don't bet the day on it.
  • WHOQA owns the test strategy: writes the TDD cases on planning day (per the hackathon rules), owns the smoke script, and hands failing tests to devs as the ticket acceptance criteria.
  • DECIDEHow is a CFML change verified before merge? (The repo's /bis-build self-test flow + dev-instance checking is the existing answer — confirm it works from a worktree.)

CAgent orchestration fits the 2:30–3:30 block · these produce your AGENTS.md

  • MUSTCompose, don't replace. bistrainerdev already has a binding CLAUDE.md (hard stops, /bis-plan→/bis-build, design-system rules, dev-DB read-only). Your hackathon AGENTS.md section is APPENDED to that — Switchboard protocol + team decisions. If a hackathon rule conflicts with a repo hard-stop, the hard-stop wins.
  • MUSTWire the knowledge tools: every agent seat gets the codegraph MCP and the BIS knowledge base (bis-knowledgebase/ + ask-BIS). These are the antidote to LLMs being weaker at CFML — agents verify against the graph and the KB instead of hallucinating CFML idioms. Also pin in AGENTS.md: Adobe CF 2023, FW/1, house idiom, and 2–3 anchor files per module ("write like this file").
  • DECIDEGate 2: keep the built-in ask-in-chat doorman on claims/dones (default), or "gate2": "off" for autonomous claiming?
  • DECIDEWIP limit: one claim per person at a time — keep, change, or drop?
  • WHOBacklog sweeper (the Gate 1 owner): named person + cadence (suggest :00 and :30). Agents propose; this person promotes what's real, kills junk, and can assign at promote time — which is also your counterweight to proposer bias.
  • DECIDEProposer bias policy: does the sweeper assign tickets to spread context, or laissez-faire (proposer claims their own)?
  • DECIDETicket template for promotion: what must a ticket contain before the sweeper promotes it (acceptance criteria, files to touch, blocked-by)?
  • MUSTDisclosure round: everyone skims their personal global CLAUDE.md / rules for anything that touches shared code style, commits, or refactoring habits — say it out loud. Personal agent config is invisible team state.

DMerge & branches captain's domain

  • MUSTEach team works on its own master branch in bistrainerdev; per-ticket worktrees branch off it and merge back per-ticket (small, continuous), not end-of-day. Set .swb.json defaultBranch to your team branch.
  • WHOThe AI captain owns merge discipline: nag per-ticket merges, run the mandatory noon checkpoint, the ~2:30 informal merge, and the 3:30 freeze. Solo-dev habit is "merge when finished" — at agent speed that is fatal; the captain's most valuable sentence all day is "merge now."
  • MUSTHumans eyeball every merge-conflict resolution. Agents resolve conflicts confidently and wrongly, especially in CFML. A conflict an agent resolved is a diff a human reads before it merges.
  • MUSTClone location: C:\dev\ (or equivalent) — never inside OneDrive-synced folders (Desktop/Documents). Worktrees + agent-speed file churn + OneDrive = lock chaos.

EReviews & the Q&A culture QA's domain

  • WHOQA coordinates Gate 3: assigns review pairs (A reviews B, B reviews C…), watches the In Review column for pile-ups, and enforces "reviewed means a human read the diff" — not a rubber stamp on agent output.
  • WHOQA stewards the ask-culture: solo devs default to letting their agent guess instead of asking. QA's job during the day: when they see a guess shipped ("assumed the schema"), route it back — "that should have been a swb ask." One enforced ask beats a wrong assumption merged.
  • MUSTNo drive-by refactors: no changes outside your ticket's declared files, however ugly the spaghetti. Agents make refactoring cheap; 2,000-line "cleanup" diffs are unreviewable. (This rule ships in the AGENTS template.)
  • MUSTNo secrets on the board: DSN strings, passwords, config values never go into Linear tickets or comments. Legacy CF has hardcoded credentials; the board is the most-read surface in the system.

FDemo & deploy OPEN — organizers must answer before the 13th

Unresolved: the day-2 agenda says "deploy to team sandbox" at the 3:30 freeze — no sandbox exists yet. The repo has no local-run story for the CF backend (no CommandBox/Docker; Adobe CF 2023 + IIS + shared dev DB), so "where does the demo run?" needs an owner. Options: (a) two dev-server sites, one per team branch; (b) each team demos from the shared dev environment switched to their branch (serialize the two demos); (c) frontend demos run local builds against the shared dev backend. Decide with whoever owns dev infra — before Friday's prep meeting, so planning day can assume the answer.
  • WHOOne named deploy owner per team — whatever the sandbox answer is, one person runs the 3:30 freeze → test → deploy sequence.
