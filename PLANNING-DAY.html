1What is "agent orchestration"? (read this once — it's the new part)

You've each worked with YOUR agent, alone. Build day is different: 8 people each running 1+ agents against the same branch at the same time. Orchestration is everything that keeps that from becoming chaos: how work gets defined, who takes what, how agents find out what everyone else did, and what "done" means when a machine wrote most of the code.

The kit gives you the plumbing: agents propose tickets into Backlog, humans promote to Todo, claims lock files + create worktrees, every prompt carries a board digest to every agent, tests gate "done", humans merge. That plumbing is neutral. What it can't decide is policy — and policy is what goes wrong. The classic failure modes, so you know what the questions below are protecting against:

  • Collision: two agents building against the same files or the same DB tables.
  • Silent divergence: two "done" tickets that don't fit together because nobody's agent knew the other's assumptions.
  • Flooding: agents proposing more work / bigger diffs than humans can review — review becomes the bottleneck, then a rubber stamp.
  • Starvation: everything depends on scaffolding; half the team idles until 10am.
  • Drift: an agent "improves" things nobody asked it to touch (drive-by refactors), or resolves a merge conflict confidently and wrongly.
  • Blind spots: a dev's agent guesses instead of asking a teammate; the guess ships.

2Codebase questions answers feed your work breakdown + dependency map

  • ANSWERWhich shared/god files does your feature touch (the big routing/engine/Application files everyone's change flows through)? Can work through them be parallelized at all — and if not, in what order do tickets go through them? One ticket at a time per such file, or something smarter you design.
  • ANSWERWhat database tables does the feature touch, and who else's tickets touch the same ones? How do schema changes get made and applied during the hackathon — by whom, batched when? NAME one DB-change owner.
  • ANSWERWhat must exist FIRST (schema, contracts, shared components, fixtures) before parallel lanes can branch? Who builds that scaffolding at 8:15, and what does everyone else do meanwhile? Exit rule: every person leaves today with ≥1 claimable ticket that does NOT depend on the scaffolding.
  • DECIDEWhich parts are frontend (React — builds and tests locally in a worktree) vs CFML (needs the dev environment to verify)? Does that change who takes what?
  • ANSWERHow does a change actually get RUN and looked at during build day — what's your team's edit → see-it-working loop, and how long does one lap take? (If the honest answer is "merge and check the dev server," plan your merge cadence around that.)

3Verification questions your done-gate depends on these

The kit's swb done refuses to finish a ticket unless YOUR team's testCommand passes. You define that command. If you don't define a real one, your done-gate is decoration — and the judging rubric has a 15% automated-testing line.
  • ANSWERWhat is your team's testCommand? What do you already run today that could gate a ticket (frontend build/lint/tests? a smoke script hitting endpoints? something else you know about that we don't)?
  • ANSWERHow is a CFML change verified before it merges — concretely, step by step?
  • NAMEQA owns the test strategy: TDD cases written today (per the hackathon rules), handed to devs as ticket acceptance criteria. Which tickets get real tests vs smoke-only?

4Orchestration questions answers go straight into your AGENTS.md

  • NAMEWho sweeps Backlog (Gate 1), on what cadence? Agents propose; this person promotes what's real, kills junk. If nobody owns this, agent-proposed work rots and the pipeline starves.
  • DECIDEDoes the sweeper ASSIGN tickets at promote time, or does whoever wants one claim it? (Watch the bias: whoever proposed a ticket tends to claim it — agent-chosen work with one human click between. Fine if you choose it; bad as drift.)
  • DECIDEGate 2 — the kit's built-in "agent must ask you in chat before claiming/finishing" doorman: keep ON (default) or set "gate2":"off" for self-serve claiming? It works even with permissions skipped; switching it off is legitimate — say it out loud.
  • DECIDEWIP limit: one claimed ticket per person at a time? (It's a norm, not code — the board makes hoarding visible; you enforce it or drop it.)
  • DECIDETicket size and template: what must a ticket contain before the sweeper promotes it (acceptance criteria, files to touch, blocked-by)? How small is a ticket — 30–90 min of agent work is the floor's suggestion.
  • DECIDESubagents / goal loops / ultracode: allowed? The floor's rule is "the claim is the unit" — whatever runs inside one claim is that dev's business, subagents never write to the board. Set your token discipline (the hackathon rules already say no spawning 100 agents).
  • DECIDECodex seats: run Codex through Claude Code's /codex plugin (full digest awareness) or bare CLI (agent must run swb sync at task start)? Who on your team is which?
  • ANSWERDisclosure round: everyone skims their personal global CLAUDE.md/rules for anything touching shared code style, commits, or refactoring habits — say it out loud. Personal agent config is invisible team state.

5Merge & review questions captain + QA domains

  • NAMEThe AI captain owns merge discipline. What's your cadence — per-ticket merges to the team branch? The noon checkpoint is mandatory (rules); what else? At agent speed, "merge when finished" is how a day dies.
  • ANSWERWho eyeballs merge-conflict resolutions? (Agents resolve conflicts confidently and wrongly; a conflict an agent resolved should be a diff a human reads.)
  • NAMEQA coordinates Gate 3 (review): assign review pairs NOW (A reviews B, B reviews C…). What's your bar for "reviewed" — a human read the diff, or something more?
  • DECIDEDrive-by refactors: the template forbids changes outside a ticket's declared files. Keep that, or define your own cleanup policy?
  • NAMEQA also stewards ask-culture: when a guess ships ("assumed the schema"), route it back — "that should have been a swb ask." Solo habit is guessing; team habit has to be asking.
  • ANSWERSecrets: agents must never paste connection strings/passwords/config values into tickets. Where DO config facts live for your team if an agent needs one?
